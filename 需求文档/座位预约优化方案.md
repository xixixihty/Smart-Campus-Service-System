## 1. 模块现状与问题分析

### 1.1 当前实现分析

当前 `SeatReservationServiceImpl` 位于 [SeatReservationServiceImpl.java](file:///d:/gitDemo/smart_campus/src/main/java/com/hxq/smart_campus/service/impl/SeatReservationServiceImpl.java)，核心流程如下：

```
客户端请求 → Controller → Service（直接操作MySQL）→ 返回结果
```

**现有功能清单**：

| 功能 | 接口 | 实现方式 |
| :--- | :--- | :--- |
| 预约座位 | POST `/api/seat-reservations/user` | 直接查DB冲突检测 + INSERT |
| 取消预约 | DELETE `/api/seat-reservations/user/{id}` | 直接UPDATE状态 |
| 签到 | POST `/api/seat-reservations/user/{id}/check-in` | 直接UPDATE状态 |

| 预约列表 | GET `/api/seat-reservations/user` | 直接SELECT分页 |

### 1.2 问题清单

| 编号 | 问题 | 影响范围 | 严重程度 | 根因 |
| :--- | :--- | :--- | :--- | :--- |
| P1 | 无分布式锁 | 高并发下同一座位被重复预约 | **严重** | 仅依赖DB行锁，跨实例无效 |
| P2 | 无缓存层 | 每次查询直达DB，响应慢 | **严重** | 未引入Redis作为查询缓存 |
| P3 | 冲突检测在应用层 | 存在TOCTOU竞态条件 | **严重** | SELECT和INSERT非原子操作 |
| P4 | 同步写DB | 事务持锁时间长，吞吐量低 | **高** | 无异步落库机制 |
| P5 | 无布隆过滤器 | 无效座位ID穿透到DB | **中** | 缺少前置校验层 |
| P6 | 无限流保护 | 突发流量可能击垮DB | **高** | 未集成Sentinel |
| P7 | 无状态机约束 | 状态流转可能异常 | **中** | 代码中if-else判断分散 |
| P8 | 无超时自动释放 | 预约未签到占用资源 | **中** | 缺少定时任务扫描 |

## 3. 通用架构模板映射

### 3.1 选课模板 → 座位预约映射

本方案完全遵循选课模块的通用架构模板，仅替换资源类型和防重逻辑：

| 模板层 | 选课模块 | 座位预约模块 |
| :--- | :--- | :--- |
| **前置过滤** | 课程ID → 布隆过滤器 | 座位ID → 布隆过滤器 |
| **原子操作** | `course:stock:{id}` 库存扣减 | `seat:schedule:{seatId}:{date}` 时间片ZSet |
| **防重逻辑** | `SISMEMBER` 防重选 | `ZRANGEBYSCORE` 检测时间冲突 |
| **兜底机制** | ZSet候补队列（满员排队） | 时间段冲突直接拒绝，无需候补 |
| **异步持久化** | MQ落库 | MQ落库 |
| **流量控制** | Sentinel限流 | Sentinel限流 |

### 3.2 通用架构模式

```
1. 【前置过滤】布隆过滤器 → 不存在直接拒绝
2. 【原子操作】Lua脚本 → 冲突检测+写入一步完成
3. 【异步持久化】MQ → Redis立即返回，MQ慢慢写DB
4. 【兜底机制】冲突直接拒绝，无需排队
5. 【流量控制】Sentinel → 接口限流+熔断降级
6. 【定时对账】比较Redis和MySQL数据，发现不一致则修复
```

### 3.3 座位预约 vs 选课差异说明

座位预约与选课的核心差异在于**资源模型**不同：

| 维度 | 选课 | 座位预约 |
| :--- | :--- | :--- |
| **资源粒度** | 课程有固定容量（整数库存） | 座位按时间片分割（无库存概念） |
| **冲突检测** | 检查是否已选过该课 | 检查该时段是否已被预约 |
| **候补机制** | 满员时排队等候退课 | 冲突直接拒绝，不排队 |
| **Redis核心结构** | String（库存）+ Set（已选） | ZSet（时间片） |

#### 5.1.3 新增表：seat_reservation_outbox（本地消息表）

```sql
CREATE TABLE seat_reservation_outbox (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    reservation_no VARCHAR(32) NOT NULL,
    event_type VARCHAR(32) NOT NULL COMMENT 'RESERVED/CANCELLED/CHECKED_IN/CHECKED_OUT',
    payload JSON NOT NULL,
    status VARCHAR(16) NOT NULL DEFAULT 'PENDING' COMMENT 'PENDING/PUBLISHED/FAILED',
    retry_count INT NOT NULL DEFAULT 0,
    max_retry INT NOT NULL DEFAULT 3,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status_created (status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 8. 业务流程逻辑

### 8.1 预约座位流程

```
1. 请求到达 → Sentinel限流检查
2. 布隆过滤器校验 seatId 是否存在
   ├─ 不存在 → 直接返回 4101
   └─ 存在 → 继续
3. 执行 Lua 脚本 reserve_seat.lua
   ├─ 返回 0（时间冲突）→ 返回 4102
   └─ 返回 1（预约成功）→ 继续
4. 发送 MQ 消息异步落库
5. 返回 200 OK
```

### 8.2 取消预约流程

```
1. 请求到达 → Sentinel限流检查
2. 执行 Lua 脚本 cancel_reservation.lua
   ├─ 返回 -1（预约不存在）→ 返回 4103
   └─ 返回 1（取消成功）→ 继续
3. 发送 MQ 消息异步落库
4. 返回 200 OK
```

### 8.3 签到流程

```
1. 请求到达 → Sentinel限流检查
2. 校验预约记录状态是否为"待签到"
   ├─ 否 → 返回 4104
   └─ 是 → 继续
3. 更新 Redis 座位状态
4. 发送 MQ 消息异步落库
5. 返回 200 OK
```

### 8.4 数据预热流程

```
每日凌晨 / 开馆前：
1. 从 MySQL 查询所有座位信息
2. 初始化布隆过滤器：添加所有有效 seatId
3. 初始化当日空闲座位集合：
   SADD seat:available:{today} {所有座位ID}
```

## 9. Lua脚本设计

### 9.1 预约座位 Lua 脚本

**文件**：`src/main/resources/lua/reserve_seat.lua`

```lua
-- reserve_seat.lua
-- KEYS[1]: seat:schedule:{seatId}:{date}  时间片ZSet
-- KEYS[2]: seat:available:{date}          空闲座位Set
-- KEYS[3]: seat:reserved:{date}           已预约Set
-- ARGV[1]: startScore                    起始分钟数
-- ARGV[2]: endScore                      结束分钟数
-- ARGV[3]: seatId
-- ARGV[4]: userId
-- ARGV[5]: reservationNo                 预约流水号

-- 1. 时间冲突检测：检查该时段是否已被预约
local existing = redis.call('ZRANGEBYSCORE', KEYS[1], ARGV[1], ARGV[2], 'LIMIT', 0, 1)
if #existing > 0 then
    return 0  -- 冲突
end

-- 2. 无冲突，写入时间片
-- member格式: 结束分钟数_用户ID_预约流水号
local member = ARGV[2] .. '_' .. ARGV[4] .. '_' .. ARGV[5]
redis.call('ZADD', KEYS[1], ARGV[1], member)

-- 3. 更新座位集合
redis.call('SREM', KEYS[2], ARGV[3])
redis.call('SADD', KEYS[3], ARGV[3])
return 1  -- 预约成功
```

### 9.2 取消预约 Lua 脚本

**文件**：`src/main/resources/lua/cancel_reservation.lua`

```lua
-- cancel_reservation.lua
-- KEYS[1]: seat:schedule:{seatId}:{date}  时间片ZSet
-- KEYS[2]: seat:available:{date}          空闲座位Set
-- KEYS[3]: seat:reserved:{date}           已预约Set
-- ARGV[1]: seatId
-- ARGV[2]: userId
-- ARGV[3]: startScore                    起始分钟数

-- 1. 查找并移除该用户的时间片记录
-- member格式: 结束分钟数_用户ID_预约流水号
-- 通过score匹配起始时间，member模糊匹配用户ID
local members = redis.call('ZRANGEBYSCORE', KEYS[1], ARGV[3], ARGV[3])
local removed = false
for i, member in ipairs(members) do
    -- member格式: endScore_userId_reservationNo
    local _, uid = member:match('^(%d+)_(%d+)_')
    if uid == ARGV[2] then
        redis.call('ZREM', KEYS[1], member)
        removed = true
        break
    end
end

if not removed then
    return -1  -- 未找到预约记录
end

-- 2. 更新座位集合
redis.call('SREM', KEYS[3], ARGV[1])
redis.call('SADD', KEYS[2], ARGV[1])
return 1  -- 取消成功
```

@RabbitListener(queues = "seat-reservation-queue")
public void handleReservation(SeatReservationMessage msg, Channel channel,
                              @Header(AmqpHeaders.DELIVERY_TAG) long tag) {
    try {
        if (seatReservationMapper.existsByReservationNo(msg.getReservationNo())) {
            channel.basicAck(tag, false);
            return;
        }
        seatReservationMapper.insert(toEntity(msg));
        channel.basicAck(tag, false);

        log.error("消费预约消息失败, reservationNo={}", msg.getReservationNo(), e);
        channel.basicNack(tag, false, false);
    }
}

## 11. 高并发处理策略

### 11.1 并发控制方案

| 层级 | 方案 | 说明 |
| :--- | :--- | :--- |
| **入口层** | Sentinel限流 | 预约接口QPS限制3000，超出快速失败 |

| **缓存层** | Redis Cluster | 分片存储，分散热点Key压力 |
| **数据层** | 读写分离 | 写走Master，读走Slave |

### 11.2 热点座位处理

1. **Caffeine本地缓存**：座位基本信息缓存5分钟，减少Redis调用
2. **空闲座位集合**：每日预热到Redis Set，查询无需扫描DB

## 13. 容错与降级机制

### 13.1 故障场景与应对

| 故障场景 | 应对策略 | 恢复方式 |
| :--- | :--- | :--- |
| **Redis不可用** | 降级为直接操作MySQL | Redis恢复后，定时任务对账修复 |
| **RabbitMQ不可用** | Outbox本地消息表兜底 | MQ恢复后，定时扫描PENDING消息重发 |
| **MySQL不可用** | 预约返回降级提示 | 依赖Redis继续服务，MySQL恢复后异步补录 |
| **Sentinel触发限流** | 返回"系统繁忙，请稍后重试" | 流量下降后自动恢复 |
