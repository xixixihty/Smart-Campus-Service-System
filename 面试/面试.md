### 2.3 JWT 认证与安全

**项目实现链路：**

```
请求 → OncePerRequestFilter → 提取 Authorization Header → 解析 Bearer Token
→ JwtUtils.parseToken() → Claims 提取用户信息 → SecurityContext.setUserInfo()
→ filterChain.doFilter() → finally: SecurityContext.clear()
```

**关键设计：**
- `SecurityContext` 使用 `ThreadLocal<LoginUserInfo>` 存储用户信息，请求结束自动清理
- `JwtUtils` 使用 HMAC-SHA256 签名，jjwt 0.12.x API
- `BCryptUtils` 使用 BCrypt 哈希（cost=12），哈希长度校验防截断攻击
- 白名单路径：`/api/auth/login`、Swagger 路径跳过认证

**面试高频问题：**
1. **JWT 的组成结构？** → Header（算法）+ Payload（Claims）+ Signature（签名）
2. **JWT 如何防止篡改？** → 服务端用密钥验证签名，任何修改都会导致签名不匹配
3. **JWT 过期如何处理？** → 短期 Access Token + 长期 Refresh Token 双令牌机制
4. **ThreadLocal 内存泄漏风险？** → 必须 finally 块中 `remove()`，线程池场景尤其重要
5. **BCrypt 为什么比 MD5/SHA 安全？** → 自带盐值 + 可调节计算成本（cost factor）+ 抗彩虹表

### 2.4 Redis 缓存架构

**项目中的多级缓存策略：**

```
请求 → Caffeine 本地缓存（5分钟过期）
     → Redis 分布式缓存（1天过期）
     → MySQL 数据库
```

**Redis 使用场景汇总：**

| 场景 | Key 设计 | 数据结构 |
|------|----------|----------|
| 课程库存 | `course:stock:{courseId}` | String（计数器） |
| 用户已选课程 | `course:selected:{userId}` | Set |
| 选课候补队列 | `course:waiting:{courseId}` | ZSet（按时间戳排序） |
| 课程详情缓存 | `course:info:{courseId}` | String（JSON） |
| 图书库存 | `book:stock:{bookId}` | String |
| 用户已借图书 | `book:borrowed:{userId}` | Set |
| 借阅候补队列 | `book:waiting:{bookId}` | ZSet |
| 座位预约时间表 | `seat:schedule:{seatId}:{date}` | ZSet（按分钟数排序） |
| 可用座位集合 | `seat:available:{date}` | Set |
| 已预约记录 | `seat:reserved:{date}` | Hash |

**面试高频问题：**
1. **缓存穿透、击穿、雪崩的区别和解决方案？**
   - 穿透：查不存在的数据 → **布隆过滤器**（项目中正是这样做的）
   - 击穿：热点 key 过期 → 互斥锁 / 逻辑过期
   - 雪崩：大量 key 同时过期 → TTL 加随机值
2. **Redis 数据结构及应用场景？** → String（计数/缓存）、Hash（对象）、List（队列）、Set（去重）、ZSet（排行榜/延迟队列）
3. **Redis 持久化 RDB vs AOF？** → RDB 快照、AOF 日志，混合持久化
4. **Redis 过期删除策略？** → 惰性删除 + 定期删除
5. **多级缓存一致性如何保证？** → 先更新 DB，再删除缓存（Cache Aside 模式）

### 2.5 布隆过滤器（Bloom Filter）

**项目中的实现：**

```java
// 初始化：预期 5000 个元素，误判率 3%
courseFilter.tryInit(5000L, 0.03);

// 预热：从 DB 加载所有课程 ID
courseIds.forEach(id -> courseFilter.add("course:" + id));

// 前置校验：快速过滤不存在的课程
courseFilter.contains("course:" + courseId);
```

**应用场景：**
- 课程存在性校验（`bf:courses`）
- 座位存在性校验（`bf:seats`）
- 图书存在性校验（`bf:books`）
- 用户存在性校验（`bf:users`）

**面试高频问题：**
1. **布隆过滤器的原理？** → 多个哈希函数 + 位数组，可能存在假阳性（误判存在），不存在假阴性
2. **如何选择哈希函数数量和位数组大小？** → 根据预期元素数量 n 和误判率 p 计算
3. **布隆过滤器不支持删除，如何解决？** → 计数布隆过滤器（Counting Bloom Filter）

### 2.6 Redisson 分布式锁

**项目中的实现：**

```java
RLock lock = redissonClient.getLock("lock:course:" + courseId);
try {
    if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {  // 等待3秒，锁10秒自动释放
        // 业务逻辑
    }
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

**面试高频问题：**
1. **Redisson 分布式锁的 Watchdog 机制？** → 自动续期，默认每 10 秒续期到 30 秒
2. **Redis 分布式锁 vs Zookeeper 分布式锁？** → Redis：AP 模型，性能高；ZK：CP 模型，一致性更强
3. **Redlock 算法是什么？** → 多节点 Redis 分布式锁，过半节点加锁成功才算成功

### 2.7 RabbitMQ 消息队列

**项目中的使用：**

| 场景 | Exchange | Queue | 说明 |
|------|----------|-------|------|
| 选课异步落库 | `course-selection-exchange` | `course-selection-queue` | Topic 模式 |
| 借阅异步落库 | `borrow-exchange` | `borrow-queue` | 异步处理 |
| 座位预约异步落库 | `seat-reservation-exchange` | `seat-reservation-queue` | 异步处理 |

**关键设计：**
- 手动 ACK 模式：`acknowledge-mode: manual`
- 死信队列：`x-dead-letter-exchange` 处理失败消息
- 并发消费：`concurrency: 10, max-concurrency: 50`
- 幂等性保证：通过 `reservation_no` 唯一索引防止重复消费

**面试高频问题：**
1. **RabbitMQ 如何保证消息不丢失？** → 生产者 Confirm、持久化、消费者手动 ACK
2. **如何保证消息幂等性？** → 唯一业务 ID + 数据库唯一索引（项目中的 `reservation_no`）
3. **死信队列的三种触发方式？** → 消息被拒绝（reject/nack）、消息 TTL 过期、队列达到最大长度
4. **RabbitMQ vs Kafka 选型？** → RabbitMQ：低延迟、复杂路由；Kafka：高吞吐、日志收集、流处理

### 2.8 Redis + Lua 脚本原子操作

**项目中的 Lua 脚本（select_course.lua）：**

```lua
-- 原子操作：检查已选 → 检查库存 → 扣减库存 → 添加到已选集
local selected = redis.call('SISMEMBER', selected_key, course_id)
if selected == 1 then return -2 end  -- 已选

local stock = tonumber(redis.call('GET', stock_key) or "0")
if stock <= 0 then
    redis.call('ZADD', waiting_key, timestamp, user_id)  -- 加入候补
    return -1
end

redis.call('DECR', stock_key)
redis.call('SADD', selected_key, course_id)
return 1
```

**Lua 脚本清单：**
- `select_course.lua` — 选课（检查已选 → 扣库存 → 加入已选）
- `drop_course.lua` — 退课（检查已选 → 加库存 → 从候补队列补位）
- `borrow_book.lua` — 借书（检查已借 → 扣库存 → 加入已借）
- `return_book.lua` — 还书（检查已借 → 加库存 → 从候补队列补位）
- `reserve_seat.lua` — 预约座位（检查时间冲突 → 记录预约）
- `cancel_reservation.lua` — 取消预约（移除预约记录 → 释放时间段）

**面试高频问题：**
1. **为什么用 Lua 脚本而不是 Redis 事务？** → Lua 脚本原子性更强，支持条件判断和复杂逻辑
2. **Redis 事务（MULTI/EXEC）的局限性？** → 不支持回滚，不支持条件判断，命令间无法获取前一个命令的结果
3. **Lua 脚本在 Redis 中执行的原子性保证？** → Redis 单线程执行 Lua 脚本，执行期间不处理其他命令

### 4.2 数据库设计规范

- 软删除：`is_deleted` 标记
- 状态字段使用有意义的字符串（如 `"待签到"`、`"使用中"`、`"已完成"`）
- 外键关系在应用层维护，数据库层使用索引优化查询

## 五、高并发场景设计（核心亮点）

### 5.1 选课系统高并发方案

```
用户请求 → 布隆过滤器（校验课程存在）
        → 时间冲突校验
        → Redisson 分布式锁（课程级别）
        → Redis Lua 脚本（原子扣库存）
        → RabbitMQ 异步落库
```

**关键设计决策：**
- **库存放在 Redis**：避免 MySQL 行锁竞争
- **Lua 脚本原子操作**：保证"检查 → 扣库存 → 记录"的原子性
- **候补队列（ZSet）**：库存不足时按时间戳排队，退课时自动补位
- **异步落库**：选课成功后通过 MQ 异步写入 MySQL，不阻塞用户响应

### 5.2 座位预约系统

```
用户请求 → 布隆过滤器（校验座位存在）
        → Redis Lua 脚本（检查时间冲突 → 记录预约）
        → RabbitMQ 异步落库
```

**时间冲突检测算法：**
- 将时间转换为分钟数（`hour * 60 + minute`）
- 使用 Redis ZSet 存储已预约时间段
- Lua 脚本中检查 ZSet 是否有重叠区间

### 5.3 图书借阅系统

与选课系统类似，使用 Redis 库存 + Lua 脚本 + 候补队列 + MQ 异步落库。
