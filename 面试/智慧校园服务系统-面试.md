## 一、Java 基础

### 简历涉及知识点

- ThreadLocal（用户上下文存储）
- HashMap / ConcurrentHashMap
- 集合框架（List、Set、Map、ZSet 的应用场景）
- 异常处理（自定义异常、全局异常处理）
- 泛型（`Result<T>` 统一响应体）
- Lambda 表达式 / Stream API

### 面试问题与答案

**Q1：ThreadLocal 的原理是什么？有什么注意事项？**

> ThreadLocal 为每个线程提供独立的变量副本。每个 Thread 内部有一个 ThreadLocalMap，Key 是 ThreadLocal 的弱引用，Value 是存储的值。
>
> **注意事项：**
> - 使用完必须调用 `remove()` 清理，否则会造成内存泄漏（尤其在线程池场景）
> - 项目中在 `JwtAuthenticationFilter` 的 `finally` 块中调用 `SecurityContext.clear()` 正是这个原因

**Q2：HashMap 的底层原理？**

> JDK8 中 HashMap 采用 **数组 + 链表 + 红黑树** 结构：
> - 默认容量 16，负载因子 0.75，扩容为原来的 2 倍
> - 链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转为红黑树
> - 红黑树节点数 ≤ 6 时，退化为链表
> - put 过程：计算 hash → 定位数组下标 → 头插/尾插 → 判断是否需要扩容/树化

**Q3：HashMap 和 ConcurrentHashMap 的区别？**

> - HashMap：线程不安全，多线程 put 可能导致死循环（JDK7）或数据覆盖
> - ConcurrentHashMap（JDK8）：CAS + synchronized 锁链表/红黑树头节点，线程安全且并发度高
> - JDK7 的 ConcurrentHashMap 使用分段锁（Segment），JDK8 改为更细粒度的节点锁

**Q4：项目中用到了哪些集合类？分别什么场景？**

> | 集合类 | 项目场景 |
> |--------|----------|
> | `List` | 批量删除 ID 列表、查询结果列表 |
> | `Set` | Redis 中存储用户已选课程（`SADD`/`SISMEMBER`） |
> | `Map` / `HashMap` | JWT Claims 存储、请求参数封装 |
> | `ZSet`（Redis） | 选课候补队列（按时间戳排序）、座位预约时间段 |

**Q5：`synchronized` 和 `ReentrantLock` 的区别？**

> | 特性 | synchronized | ReentrantLock |
> |------|-------------|---------------|
> | 实现 | JVM 层面，关键字 | JDK 层面，API |
> | 锁释放 | 自动释放 | 必须 finally 中 unlock |
> | 可中断 | 不支持 | `lockInterruptibly()` |
> | 公平锁 | 非公平 | 可选公平/非公平 |
> | 条件变量 | 单一 Condition | 可绑定多个 Condition |
> | 尝试加锁 | 不支持 | `tryLock()` 支持超时 |

## 二、Spring Boot / Spring 框架

### 简历涉及知识点

- Spring Boot 3 自动配置
- Spring IoC / DI（`@Autowired`、`@RequiredArgsConstructor`）
- Spring 声明式事务（`@Transactional`）
- `@RestControllerAdvice` 全局异常处理
- `@PostConstruct` 初始化回调
- `OncePerRequestFilter` 过滤器
- `@Configuration` + `@Bean` 配置类
- `@Value` 配置注入
- SpringDoc OpenAPI（Swagger 3）

### 面试问题与答案

**Q1：Spring Boot 自动配置原理？**

> 核心入口是 `@SpringBootApplication` 中的 `@EnableAutoConfiguration`：
> 1. 通过 `@Import(AutoConfigurationImportSelector.class)` 加载自动配置类
> 2. 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（Spring Boot 3.x）
> 3. 结合 `@Conditional` 系列注解（`@ConditionalOnClass`、`@ConditionalOnMissingBean` 等）按条件装配
> 4. 例如：classpath 有 `RedisTemplate` 则自动配置 Redis，有 `DataSource` 则自动配置数据源

**Q2：Spring Bean 的生命周期？**

> 实例化 → 属性注入（`@Autowired`）→ Aware 接口回调 → `@PostConstruct` 初始化 → `InitializingBean.afterPropertiesSet()` → Bean 就绪 → `@PreDestroy` 销毁 → `DisposableBean.destroy()`
>
> 项目中 `JwtUtils` 的 `@PostConstruct` 方法用于将 `@Value` 注入的非静态字段赋值给静态变量。

**Q3：`@Transactional` 失效的场景有哪些？**

> 1. **非 public 方法**：Spring AOP 代理只能拦截 public 方法
> 2. **同类方法调用（this 调用）**：不经过代理对象，事务不生效
> 3. **异常被 catch 未抛出**：事务只对未捕获的 RuntimeException 和 Error 回滚
> 4. **数据库引擎不支持事务**：如 MyISAM
> 5. **多线程环境**：事务绑定在线程上，新线程不在同一事务中

**Q4：Spring 事务传播行为有哪些？**

> | 传播行为 | 说明 |
> |----------|------|
> | REQUIRED（默认） | 有事务则加入，无则新建 |
> | REQUIRES_NEW | 总是新建事务，挂起当前事务 |
> | NESTED | 嵌套事务，内层回滚不影响外层 |
> | SUPPORTS | 有事务则加入，无则非事务运行 |
> | MANDATORY | 必须在事务中，否则抛异常 |
> | NOT_SUPPORTED | 非事务运行，挂起当前事务 |
> | NEVER | 必须在非事务中，否则抛异常 |

**Q5：`@RestControllerAdvice` 的作用？**

> `@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`，用于全局异常处理：
> - 集中处理所有 Controller 抛出的异常
> - 返回 JSON 格式的统一响应体
> - 项目中按异常类型分级处理：`NotFoundException` → 404、`BusinessException` → 400、`Exception` → 500

**Q6：Spring Boot 3.x 相比 2.x 的主要变化？**

> - **Jakarta EE 迁移**：`javax.*` → `jakarta.*`（如 `javax.servlet` → `jakarta.servlet`）
> - **Java 17 基线**：最低要求 Java 17
> - **AOT 编译支持**：提前编译，提升启动速度
> - **虚拟线程支持**：Java 21 虚拟线程
> - **GraalVM 原生镜像**：更好的原生编译支持
> - **Observability**：内置 Micrometer 可观测性

## 三、MyBatis

### 简历涉及知识点

- MyBatis 原生 XML Mapper
- PageHelper 分页插件
- 动态 SQL（`<if>`、`<where>`、`<foreach>`）
- 批量操作
- 多表联合查询映射到 VO
- 下划线转驼峰自动映射

### 面试问题与答案

**Q1：`#{}` 和 `${}` 的区别？**

> - `#{}`：预编译占位符，生成 `?`，防止 SQL 注入，只能替换值
> - `${}`：字符串直接拼接，可能引发 SQL 注入，用于动态表名/列名/ORDER BY 等场景
>
> **项目中**：所有用户输入的参数都使用 `#{}`，确保安全。

**Q2：MyBatis 一级缓存和二级缓存？**

> - **一级缓存**：SqlSession 级别，默认开启，同一 SqlSession 内相同查询只执行一次 SQL
> - **二级缓存**：Mapper 级别，需手动配置 `<cache/>`，跨 SqlSession 共享，通常配合 Redis 等分布式缓存使用
> - 项目中主要使用 Redis + Caffeine 做缓存，未使用 MyBatis 二级缓存

**Q3：PageHelper 分页原理？**

> PageHelper 通过 MyBatis 拦截器（Interceptor）实现：
> 1. `PageHelper.startPage(pageNum, pageSize)` 将分页参数存入 ThreadLocal
> 2. 拦截器在 SQL 执行前拦截，自动追加 `LIMIT ?, ?`
> 3. 同时执行 `COUNT(*)` 查询获取总数
> 4. 将查询结果封装为 `PageInfo` 对象

**Q4：MyBatis 和 MyBatis-Plus 的区别？**

> - MyBatis：需手写 SQL/XML，灵活度高，适合复杂查询
> - MyBatis-Plus：内置 CRUD（`BaseMapper`）、条件构造器（`LambdaQueryWrapper`）、分页插件更简洁
> - 项目中选用原生 MyBatis，因为复杂多表查询需要精确控制 SQL

## 四、MySQL / 数据库

### 简历涉及知识点

- 数据库索引设计（单列索引、复合索引、唯一索引）
- 最左前缀原则
- 索引失效场景
- EXPLAIN 执行计划分析
- 乐观锁 vs 悲观锁
- 数据库设计规范（主键、时间字段、软删除）

### 面试问题与答案

**Q1：项目中索引是如何设计的？以排课表为例。**

> 排课表 `course_schedule` 的索引设计：
> - `idx_course_schedule_semester`：按学期查询
> - `idx_course_schedule_teacher`：按教师查询 + 冲突检测
> - `idx_course_schedule_classroom`：按教室查询 + 冲突检测
> - `idx_course_schedule_time(week_day, start_section, end_section)`：时间冲突检测（复合索引）
> - `idx_course_schedule_semester_teacher(semester_id, teacher_id)`：列表查询优化（联合索引）
> - `idx_seat_reservation_no`：唯一索引，MQ 幂等性保证

**Q2：什么是最左前缀原则？**

> 复合索引 `(A, B, C)` 会创建三个索引：`(A)`、`(A, B)`、`(A, B, C)`。
> - `WHERE A = ? AND B = ?` → 命中索引
> - `WHERE A = ? AND C = ?` → 只用到 A，C 不走索引（跳过了 B）
> - `WHERE B = ? AND C = ?` → 不走索引（缺少最左列 A）
> - `WHERE A > ? AND B = ?` → A 走范围索引，B 不走（范围查询后的列失效）

**Q3：索引失效的常见场景？**

> 1. LIKE `%keyword`（前导模糊）
> 2. OR 条件中有一个列无索引
> 3. 对索引列使用函数：`WHERE DATE(create_time) = '2026-01-01'`
> 4. 隐式类型转换：`WHERE phone = 13800138000`（phone 是 VARCHAR）
> 5. 联合索引不满足最左前缀
> 6. `!=`、`NOT IN`、`IS NULL` / `IS NOT NULL` 在某些情况下不走索引

**Q4：EXPLAIN 执行计划的关键字段？**

> | 字段 | 含义 | 关注点 |
> |------|------|--------|
> | type | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
> | key | 实际使用的索引 | 是否命中预期索引 |
> | rows | 预估扫描行数 | 越小越好 |
> | Extra | 额外信息 | Using index（覆盖索引，好）/ Using filesort（文件排序，差）/ Using temporary（临时表，差） |

**Q5：乐观锁和悲观锁的区别？项目中用到了哪种？**

> - **悲观锁**：认为冲突频繁，先加锁再操作（`SELECT ... FOR UPDATE`）
> - **乐观锁**：认为冲突不频繁，操作时检查版本号（version 字段或 CAS）
>
> 项目中：
> - 选课系统使用 **Redisson 分布式锁**（悲观锁思路），因为选课并发冲突概率高
> - 数据库层面使用唯一索引（如 `reservation_no`）防止重复插入

## 五、Redis

### 简历涉及知识点

- Redis 五种基本数据结构（String、Hash、List、Set、ZSet）
- Redis Lua 脚本原子操作
- 布隆过滤器（Bloom Filter）
- 缓存穿透/击穿/雪崩
- Redis 持久化（RDB / AOF）
- Redis 过期删除策略
- Redis 事务 vs Lua 脚本
- Redis 分布式锁（见第七章）

### 面试问题与答案

**Q1：项目中 Redis 的 Key 是如何设计的？**

> 采用 **冒号分隔的层级命名**，如：
> - `course:stock:{courseId}` — 课程库存
> - `course:selected:{userId}` — 用户已选课程集合
> - `course:waiting:{courseId}` — 选课候补队列
> - `seat:schedule:{seatId}:{date}` — 座位预约时间表
> - `book:stock:{bookId}` — 图书库存
>
> 好处：可读性强、便于按前缀批量操作、支持 Redis Cluster 的 hash tag

**Q2：缓存穿透、击穿、雪崩的区别和解决方案？**

> | 问题 | 原因 | 项目中的解决方案 |
> |------|------|-----------------|
> | **穿透** | 查询不存在的数据，请求直达 DB | **布隆过滤器**：`bf:courses`、`bf:seats`、`bf:books` 前置拦截 |
> | **击穿** | 热点 key 过期，大量请求直达 DB | **分布式锁**：Redisson 加锁，只让一个请求回源 DB |
> | **雪崩** | 大量 key 同时过期 | **多级缓存**：Caffeine 本地缓存兜底；**随机 TTL**：避免同时过期 |

**Q3：Redis 五种数据结构及其在项目中的应用？**

> | 数据结构 | 项目场景 |
> |----------|----------|
> | **String** | 课程库存计数（`DECR`）、课程详情 JSON 缓存 |
> | **Hash** | 苍穹外卖购物车（`HSET cart:userId field value`） |
> | **List** | 苍穹外卖菜品分类缓存（`LPUSH` / `LRANGE`） |
> | **Set** | 用户已选课程（`SADD`/`SISMEMBER`）、可用座位集合 |
> | **ZSet** | 选课候补队列（按时间戳排序）、座位预约时间段（按分钟数排序） |

**Q4：为什么用 Lua 脚本而不是 Redis 事务？**

> - Redis 事务（MULTI/EXEC）**不支持条件判断**，无法实现"检查库存 → 扣库存"这种逻辑
> - Redis 事务**不支持回滚**，某条命令失败不影响其他命令
> - Lua 脚本在 Redis 中**原子执行**，执行期间不处理其他命令
> - Lua 脚本支持**复杂逻辑**（if/else、循环、变量）
>
> 项目中 6 个 Lua 脚本都涉及"检查 → 判断 → 操作"的原子流程。

**Q5：Redis 持久化 RDB 和 AOF 的区别？**

> | 特性 | RDB | AOF |
> |------|-----|-----|
> | 原理 | 定时快照（二进制） | 记录每条写命令（文本） |
> | 恢复速度 | 快 | 慢（需重放命令） |
> | 数据安全 | 可能丢失最后一次快照后的数据 | 最多丢 1 秒数据 |
> | 文件大小 | 小 | 大 |
> | 推荐 | 生产环境两者结合使用 | |

**Q6：布隆过滤器的原理？项目中如何使用？**

> **原理**：使用多个哈希函数将元素映射到位数组中的多个位置。查询时如果所有位置都是 1，则元素"可能存在"；如果有任一位置为 0，则元素"一定不存在"。
>
> **特点**：可能存在假阳性（误判存在），不存在假阴性（不会漏判）。
>
> **项目使用**：
> - 启动时从 DB 加载所有课程/座位/图书 ID 到布隆过滤器
> - 请求进来先查布隆过滤器，不存在直接返回，避免缓存穿透
> - 使用 Redisson 的 `RBloomFilter`，预期 5000 元素，误判率 3%

## 六、消息队列 RabbitMQ

### 简历涉及知识点

- RabbitMQ 核心概念（Exchange、Queue、Binding、Routing Key）
- 手动 ACK 机制
- 死信队列（DLX）
- 消息幂等性
- 消息可靠性（生产者 Confirm、持久化、消费者 ACK）
- 异步削峰填谷

### 面试问题与答案

**Q1：项目中 RabbitMQ 的使用场景有哪些？**

> | 场景 | Exchange | Queue | 说明 |
> |------|----------|-------|------|
> | 选课异步落库 | `course-selection-exchange` | `course-selection-queue` | Topic 模式 |
> | 借阅异步落库 | `borrow-exchange` | `borrow-queue` | 异步处理 |
> | 座位预约异步落库 | `seat-reservation-exchange` | `seat-reservation-queue` | 异步处理 |
> | 苍穹外卖订单 | 订单 Exchange | 订单 Queue | 库存扣减 |

**Q2：RabbitMQ 如何保证消息不丢失？**

> 从三个环节保证：
> 1. **生产者 → MQ**：开启 Publisher Confirm 机制，消息未到达 Broker 时重发
> 2. **MQ 内部**：队列和消息都设置为持久化（`durable=true`、`persistent`）
> 3. **MQ → 消费者**：手动 ACK 模式（`acknowledge-mode: manual`），处理成功才 `basicAck`，失败则 `basicNack` 重新入队
>
> 项目中配置了 `acknowledge-mode: manual`，消费者处理完后手动确认。

**Q3：如何保证消息消费的幂等性？**

> 项目中通过 **唯一业务 ID + 数据库唯一索引** 实现：
> - 座位预约：`reservation_no` 字段 + `UNIQUE INDEX idx_seat_reservation_no`
> - 即使 MQ 重复投递，数据库唯一索引会拒绝重复插入
> - 消费者捕获 `DuplicateKeyException` 后直接 ACK，不重复处理

**Q4：死信队列（DLX）的三种触发方式？**

> 1. **消息被拒绝**：`basicNack` / `basicReject` 且 `requeue=false`
> 2. **消息 TTL 过期**：队列设置了 `x-message-ttl`
> 3. **队列达到最大长度**：`x-max-length` 溢出
>
> 项目中配置了 `x-dead-letter-exchange`，失败消息进入死信队列，便于后续排查和人工处理。

**Q5：RabbitMQ 和 Kafka 的选型区别？**

> | 特性 | RabbitMQ | Kafka |
> |------|----------|-------|
> | 设计目标 | 消息中间件 | 分布式流平台 |
> | 吞吐量 | 万级/秒 | 百万级/秒 |
> | 延迟 | 微秒级 | 毫秒级 |
> | 消息回溯 | 不支持 | 支持（基于 offset） |
> | 适用场景 | 业务异步解耦、延迟低 | 日志收集、流处理、大数据 |
>
> 项目选 RabbitMQ 是因为业务场景需要低延迟和灵活的路由规则。

## 七、分布式锁 Redisson

### 简历涉及知识点

- Redisson 分布式锁（`RLock`）
- Watchdog 自动续期机制
- `tryLock` 等待超时 + 锁超时
- Redlock 算法

### 面试问题与答案

**Q1：项目中分布式锁的使用场景？**

> 选课系统中，对**课程级别**加分布式锁：
> ```java
> RLock lock = redissonClient.getLock("lock:course:" + courseId);
> lock.tryLock(3, 10, TimeUnit.SECONDS);  // 等待3秒，锁10秒自动释放
> ```
> 防止同一课程被超额选课。

**Q2：Redisson 的 Watchdog 机制是什么？**

> - 如果 `tryLock` 时没有指定 `leaseTime`（或设为 -1），Redisson 会启动 Watchdog
> - Watchdog 默认每 **10 秒** 续期一次，将锁过期时间续到 **30 秒**
> - 防止业务执行时间过长导致锁提前释放
> - 如果指定了 `leaseTime`（如 10 秒），则不会自动续期

**Q3：Redis 分布式锁 vs Zookeeper 分布式锁？**

> | 特性 | Redis（Redisson） | Zookeeper |
> |------|-------------------|-----------|
> | CAP | AP 模型（优先可用性） | CP 模型（优先一致性） |
> | 性能 | 高 | 较低 |
> | 实现复杂度 | 低（Redisson 封装好） | 中（需处理临时节点） |
> | 可靠性 | 主从切换可能丢锁 | 更可靠 |
> | 适用场景 | 高并发、性能敏感 | 强一致性要求 |

**Q4：Redlock 算法是什么？**

> Redlock 是 Redis 官方提出的多节点分布式锁算法：
> 1. 客户端向 N 个独立的 Redis 节点（通常 5 个）依次请求加锁
> 2. 设置超时时间远小于锁的有效时间
> 3. 当且仅当**过半节点**（N/2+1）加锁成功，才算获取锁成功
> 4. 如果获取失败，向所有节点发送解锁请求
>
> 解决了单节点 Redis 主从切换可能丢锁的问题。

## 八、多级缓存架构 Caffeine + Redis

### 简历涉及知识点

- Caffeine 本地缓存
- Redis 分布式缓存
- 三级缓存架构（Caffeine → Redis → MySQL）
- Cache Aside 模式
- 缓存一致性

### 面试问题与答案

**Q1：项目中多级缓存的架构是怎样的？**

> ```
> 请求 → Caffeine 本地缓存（5分钟过期，JVM 内存）
>      → Redis 分布式缓存（1天过期，共享）
>      → MySQL 数据库
> ```
>
> **查询流程**：
> 1. 先查 Caffeine，命中直接返回（最快）
> 2. 未命中查 Redis，命中后回填 Caffeine
> 3. 仍未命中查 MySQL，回填 Redis 和 Caffeine
>
> **更新流程**：先更新 DB，再同时删除 Caffeine 和 Redis 中的缓存。

**Q2：Caffeine 和 Redis 各自的优势？**

> | 特性 | Caffeine | Redis |
> |------|----------|-------|
> | 访问速度 | 纳秒级（内存） | 毫秒级（网络 IO） |
> | 容量 | 受 JVM 堆内存限制 | 独立部署，容量大 |
> | 共享性 | 单机，不共享 | 分布式共享 |
> | 淘汰策略 | W-TinyLFU（高命中率） | LRU/LFU/TTL |
> | 一致性 | 各节点独立，可能不一致 | 全局一致 |

**Q3：如何保证缓存和数据库的一致性？**

> 采用 **Cache Aside 模式**：
> - **读**：先读缓存，未命中读 DB，回填缓存
> - **写**：先更新 DB，再删除缓存（而非更新缓存）
>
> **为什么是删除缓存而不是更新缓存？**
> - 更新缓存可能产生并发写问题
> - 删除缓存是幂等操作
> - 懒加载思想：等下次读取时再加载最新数据
>
> 项目中 `CourseCacheService.invalidateCache()` 就是同时删除 Caffeine 和 Redis。

## 十、JWT 认证与安全

### 简历涉及知识点

- JWT 组成结构（Header + Payload + Signature）
- HMAC-SHA256 签名算法
- BCrypt 密码哈希
- OncePerRequestFilter 过滤器
- ThreadLocal 用户上下文
- Token 过期处理

### 面试问题与答案

**Q1：JWT 的组成结构？**

> JWT 由三部分组成，用 `.` 分隔：
> - **Header**：`{"alg": "HS256", "typ": "JWT"}` → Base64 编码
> - **Payload**：`{"userId": 1, "username": "admin", "exp": 1234567890}` → Base64 编码
> - **Signature**：`HMAC-SHA256(base64(Header) + "." + base64(Payload), secret)`
>
> 注意：Payload 只是 Base64 编码，**不是加密**，不要存放敏感信息！

**Q2：JWT 如何防止篡改？**

> 服务端用密钥对 Header + Payload 计算签名，任何对 Header 或 Payload 的修改都会导致签名不匹配。攻击者无法伪造签名，因为他不知道服务端的密钥。

**Q3：JWT 过期了怎么办？**

> 常见方案是 **双令牌机制**：
> - Access Token：短期有效（如 15 分钟），用于 API 请求
> - Refresh Token：长期有效（如 7 天），用于刷新 Access Token
>
> 项目中目前使用单 Token（24 小时过期），对于实习项目级别是够用的。

**Q4：BCrypt 为什么比 MD5/SHA 更安全？**

> 1. **自带盐值**：每次加密生成随机盐，相同密码产生不同哈希
> 2. **可调节成本**：cost factor（项目中设为 12），计算量指数级增长
> 3. **抗彩虹表**：盐值 + 高计算成本使彩虹表攻击不可行
> 4. **抗暴力破解**：单次哈希耗时约 0.1-0.3 秒（cost=12），大幅降低破解速度

**Q5：项目中认证流程是怎样的？**

> ```
> 1. 用户登录 → AuthService 验证用户名密码（BCrypt.checkPassword）
> 2. 验证通过 → JwtUtils.generateToken() 生成 JWT
> 3. 前端存储 Token → 每次请求通过 Authorization: Bearer xxx 携带
> 4. JwtAuthenticationFilter 拦截请求 → 解析 Token → 存入 SecurityContext（ThreadLocal）
> 5. 请求结束 → finally 块中 SecurityContext.clear() 清理
> ```

### 简历涉及知识点（苍穹外卖）

- WebSocket 协议
- 前后端长连接
- Redis 维护连接状态
- 订单状态实时推送

### 面试问题与答案

**Q1：WebSocket 和 HTTP 的区别？**

> | 特性 | HTTP | WebSocket |
> |------|------|-----------|
> | 通信方式 | 请求-响应（单向） | 全双工（双向） |
> | 连接 | 短连接（1.1 可长连接） | 长连接 |
> | 头部开销 | 大（每次请求带完整头） | 小（帧头部 2-14 字节） |
> | 适用场景 | REST API、页面加载 | 实时推送、聊天、通知 |

> 苍穹外卖中，用户下单后商家端需要实时收到新订单通知：
> 1. 前端与后端建立 WebSocket 长连接

> 3. 用户下单 → 后端通过 Redis 找到商家 Session → 推送订单消息

**Q3：WebSocket 的心跳机制是什么？**

> 长连接可能因网络问题断开，心跳机制用于检测连接是否存活：
> - 客户端定时发送 Ping 帧
> - 服务端回复 Pong 帧
> - 超时未收到 Pong 则认为连接断开，触发重连

**Q1：AOP 的原理是什么？**

**Q2：项目中 AOP 的使用场景？**

> - 事务管理（`@Transactional` 就是 AOP 实现）
> - 性能监控（方法耗时统计）

## 十三、前端技术 Vue 3 / Element Plus / ECharts

### 简历涉及知识点

- Vue 3 Composition API（`<script setup>`）
- `ref` / `reactive` 响应式数据
- Vue Router 4（路由懒加载、路由守卫）
- Axios 封装（请求/响应拦截器）
- Element Plus 组件库
- ECharts 数据可视化
- Vite 构建工具
- Pinia 状态管理

### 面试问题与答案

**Q1：Vue 3 `ref` 和 `reactive` 的区别？**

> - `ref`：包装基本类型和对象，通过 `.value` 访问/修改
> - `reactive`：包装对象，直接访问属性
> - `ref` 可以替换 `reactive`，但 `reactive` 不能用于基本类型
> - 项目中表单用 `reactive`，加载状态用 `ref`

**Q2：Vue 3 响应式原理？**

> Vue 3 使用 **Proxy** 替代 Vue 2 的 `Object.defineProperty`：
> - 可以拦截数组索引赋值和 `length` 变化
> - 可以拦截对象属性的添加和删除
> - 不需要递归遍历所有属性（惰性响应式）
> - 性能更好，支持更多数据类型

**Q3：Vue Router 路由懒加载的原理？**

> 使用动态 `import()` 语法：
> ```js
> component: () => import('@/views/teaching/CourseSchedule.vue')
> ```
> Vite/Webpack 会将动态导入的组件单独打包成 chunk，只有访问该路由时才加载，减少首屏加载时间。

**Q4：Axios 拦截器的执行顺序？**

> - **请求拦截器**：后添加的先执行（栈结构，LIFO）
> - **响应拦截器**：先添加的先执行（队列结构，FIFO）
>
> 项目中请求拦截器自动附加 Token，响应拦截器统一处理 401 跳转登录页。

**Q5：Vite 为什么比 Webpack 快？**

> - **开发环境**：Vite 使用原生 ES Module，按需编译（esbuild 预构建依赖），冷启动快
> - **HMR**：只更新修改的模块，不重新打包整个应用
> - **生产环境**：使用 Rollup 打包，Tree Shaking 更彻底
> - Webpack 需要先打包整个应用再启动开发服务器

## 十四、系统设计题

### 面试高频系统设计题

**Q1：如何设计一个秒杀系统？**

> 参考项目中选课系统的设计思路：
>
> ```
> 前端 → 按钮防重复 + 验证码
>      → Nginx 限流
>      → 网关限流（Sentinel）
>      → 布隆过滤器（校验商品存在）
>      → Redis 预减库存（Lua 脚本原子操作）
>      → 成功 → MQ 异步创建订单
>      → 失败 → 返回已售罄
> ```
>
> **关键点**：
> 1. 库存放在 Redis，避免 MySQL 行锁竞争
> 2. Lua 脚本保证"检查 → 扣库存"原子性
> 3. MQ 削峰填谷，异步落库
> 4. 前端限流 + 网关限流多层防护

**Q2：如何设计一个选课系统？**

> 就是项目中的方案：
> 1. 布隆过滤器校验课程存在
> 2. 时间冲突校验
> 3. Redisson 分布式锁（课程级别）
> 4. Redis Lua 脚本原子扣库存
> 5. 库存不足 → ZSet 候补队列
> 6. 选课成功 → RabbitMQ 异步落库
> 7. 退课 → Lua 脚本加库存 + 候补队列自动补位

**Q3：如何保证接口幂等性？**

> 1. **唯一业务 ID + 数据库唯一索引**（项目中的 `reservation_no`）
> 2. **Redis SETNX**：`SET key value NX`，只有 key 不存在才成功
> 3. **Token 机制**：先获取 Token，提交时校验 Token 并删除
> 4. **状态机**：通过业务状态判断（如订单状态从"待支付"→"已支付"）

## 十五、项目场景题

### 面试官可能问的场景题

**Q1：选课瞬间大量请求，Redis 也扛不住怎么办？**

> 多层限流策略：
> 1. **前端**：按钮防抖（点击后 3 秒内不可再点）
> 2. **Nginx**：限制单 IP 请求频率（`limit_req_zone`）
> 3. **Sentinel**：QPS 限流，超过阈值直接返回"系统繁忙"
> 4. **Redis Cluster**：分片存储，分散压力
> 5. **排队机制**：超过系统承载能力的请求进入等待队列

**Q2：MQ 消息丢失怎么办？**

> 三重保障：
> 1. **生产者 Confirm**：消息未到达 Broker 时重发
> 2. **消息 + 队列持久化**：MQ 重启不丢消息
> 3. **消费者手动 ACK**：处理成功才确认，失败则 `basicNack` 重新入队
> 4. **死信队列兜底**：多次重试失败的消息进入 DLX，人工介入

**Q3：缓存和数据库不一致怎么办？**

> 1. **Cache Aside**：先更新 DB，再删除缓存
> 2. **延迟双删**：更新 DB 前删一次缓存，更新后延迟几百毫秒再删一次
> 3. **订阅 Binlog**：通过 Canal 监听 MySQL Binlog，异步更新缓存
> 4. **设置合理 TTL**：即使不一致，过期后自动修复

**Q4：分布式锁的锁粒度如何选择？**

> 项目中选课系统使用**课程级别**的锁（`lock:course:{courseId}`）：
> - 锁粒度太粗（如全局锁）：并发性能差
> - 锁粒度太细（如用户+课程锁）：锁数量爆炸，管理复杂
> - 课程级别是合理的折中：同一课程的选课操作互斥，不同课程并行

**Q5：项目中遇到的最大挑战是什么？**

> 建议回答思路：
> 1. **选课高并发**：如何保证库存扣减的原子性和一致性
> 2. **解决方案**：Redis Lua 脚本 + Redisson 分布式锁 + 布隆过滤器 + MQ 异步落库
> 3. **效果**：保证了选课的正确性，支持候补排队，系统稳定
