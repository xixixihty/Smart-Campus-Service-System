#### P0-1. 考勤管理

##### 1.1 功能需求

| 需求编号 | 需求描述 | 涉及端 |
|----------|---------|--------|
| A1 | 教师可对某节课发起考勤（限定时间窗口内签到） | 用户端（教师） |
| A2 | 学生收到考勤通知后，在限定时间内签到/签退 | 用户端（学生） |
| A3 | 支持 GPS 定位签到（可选，后期版本）或二维码扫码签到 | 用户端 |
| A4 | 管理员/教师可按课程、班级、日期查看出勤统计 | 管理端 |
| A5 | 支持查看单个学生的全学期出勤明细 | 管理端 + 用户端 |
| A6 | 出勤类型：正常、迟到、早退、缺勤、请假（关联 leave_request） | 全端 |

##### 1.2 数据模型

**新增表: `attendance`（考勤记录表）**

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | bigint(20) | PK, AUTO_INCREMENT | 主键 |
| leave_request_id | bigint(20) | FK, NOT NULL | 请假申请ID |
| step_order | int(11) | NOT NULL | 审批步骤序号（1,2,3） |
| approver_id | bigint(20) | NOT NULL | 审批人ID |
| approver_role | varchar(20) | NOT NULL | COUNSELOR/DEAN/STUDENT_AFFAIRS |
| status | varchar(20) | DEFAULT 'PENDING' | PENDING/APPROVED/REJECTED |
| comment | varchar(500) | — | 审批意见 |
| approval_time | datetime | — | 审批时间 |

**审批流程配置（从 system_config 读取）:**
```json
{
  "leave_approval_chain": [
    { "step": 1, "role": "COUNSELOR", "name": "辅导员" },
    { "step": 2, "role": "DEAN", "name": "学院领导", "condition": "duration_days > 3" },
    { "step": 3, "role": "STUDENT_AFFAIRS", "name": "学工处", "condition": "duration_days > 7" }
  ]
}
```

**流程:**
```
学生提交请假 → 生成 leave_request
→ 生成 leave_approval_step(step=1, approver=辅导员)
→ 辅导员审批通过 → 判断是否需要下一级(duration > 3天 → 生成 step=2)
→ 学院审批通过 → 判断是否需要下一级(duration > 7天 → 生成 step=3)
→ 所有步骤通过 → leave_request.status='已批准'
```

JOIN major m ON c.major_id = m.id

public SystemHealthVO getSystemHealth() {
    return SystemHealthVO.builder()
        .jvmMemoryUsed(Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory())
        .jvmMemoryMax(Runtime.getRuntime().maxMemory())
        .cpuLoad(ManagementFactory.getOperatingSystemMXBean().getSystemLoadAverage())
        .onlineUsers(redisTemplate.opsForSet().size("online:users"))
        .redisHitRate(getRedisHitRate())
        .mqPendingCount(rabbitTemplate.execute(channel -> channel.messageCount(queueName)))
        .build();
}

## 三、模块间交互逻辑与数据流转

### 3.1 事件总线模型

以 RabbitMQ 为中心构建系统事件总线，实现模块间松耦合通信：

```
                           ┌─────────────────┐
                           │   RabbitMQ       │
                           │   Exchange       │
                           └───────┬─────────┘
                                   │
         ┌─────────────────────────┼────────────────────────┐
         │                         │                         │
    ┌────▼────┐             ┌─────▼──────┐           ┌──────▼─────┐
    │ 通知模块  │◄────────────│  考勤模块   │───────────►│  WebSocket │
    │(P0-4)   │ selection    │  (P0-1)    │ attendance │  (P1-7)   │
    └────┬────┘ success/     └────────────┘ push       └────────────┘
         │     reject/remind             
         │           
    ┌────▼────┐       ┌──────────┐       ┌──────────┐
    │ 教师/学生 │       │ 操作日志   │       │ 选课模块   │
    │  用户端   │       │  (P0-3)  │       │ （现有）   │
    └─────────┘       └──────────┘       └──────────┘
```

### 3.2 核心业务事件清单

| 事件名称 | 发布者 | 消费者 | 触发时机 | 数据内容 |
|---------|--------|---------|---------|---------|
| `selection.success` | 选课服务 | 通知模块 | 选课成功 | `{ studentId, courseName, semesterId }` |
| `selection.failed` | 选课服务 | 通知模块 | 选课失败 | `{ studentId, courseName, reason }` |
| `leave.approved` | 请假服务 | 通知模块 + 考勤模块 | 请假审批通过 | `{ studentId, startDate, endDate }` |
| `leave.rejected` | 请假服务 | 通知模块 | 请假被拒绝 | `{ studentId, reason }` |
| `borrow.overdue` | 定时任务 | 通知模块 | 借阅逾期 | `{ studentId, bookName, dueDate, overdueDays }` |
| `seat.reservation.remind` | 座位服务 | 通知模块 + WebSocket | 预约开始前30分钟 | `{ studentId, seatNo, startTime }` |
| `seat.violation.recorded` | 定时任务 | 座位模块 | 爽约记录 | `{ studentId, violationCount }` |
| `attendance.task.start` | 考勤服务 | WebSocket | 教师发起签到 | `{ taskId, courseName, studentIds }` |
| `exam.remind` | 定时任务 | 通知模块 + WebSocket | 考试前30分钟 | `{ studentId, examName, startTime, location }` |
| `score.published` | 成绩服务 | 通知模块 | 成绩发布 | `{ studentId, courseName, score }` |
| `audit.log` | AOP切面 | 操作日志模块 | 管理端写操作 | `{ userId, module, operationType, params }` |

### 3.3 关键数据流转

**考勤闭环（新模块协同最典型场景）:**
```
┌──────────────────────────────────────────────────────────────┐
│ 1. 教师发起考勤(attendance_task)                               │
│    ↓                                                          │
│ 2. RabbitMQ → WebSocket 推送给该课所有学生                     │
│    ↓                                                          │
│ 3. 学生在窗口内签到(attendance 记录)                            │
│    ├─ 正常签到: status='正常'                                  │
│    └─ 超阈值签到: status='迟到'                                │
│    ↓                                                          │
│ 4. 窗口结束 → 定时任务扫描未签到学生                             │
│    ├─ 有当日请假记录(leave_request): status='请假'             │
│    └─ 无请假: status='缺勤'                                    │
│    ↓                                                          │
│ 5. RabbitMQ 发送 attendance.result 事件                       │
│    ↓                                                          │
│ 6. 通知模块: 生成「您今日XX课程缺勤」通知给缺勤学生              │
│    辅导员: 可在考勤统计看板查看班级出勤率                        │
└──────────────────────────────────────────────────────────────┘
```

**消息系统桥接:**
```
所有自动通知生成点                  用户触达通道
┌──────────────────┐          ┌──────────────────┐
│ 选课成功/失败      │──┐       │                  │
│ 请假审批结果       │  │       │  WebSocket 实时   │──► 在线用户 → 即时弹窗
│ 预约到期提醒       │──┼───►  │  推送(P1-7)      │
│ 借阅催还          │  │       │                  │
│ 考试倒计时         │  │       ├──────────────────┤
│ 成绩发布          │──┘       │  站内信(P0-4)    │──► 在线/离线 → 消息列表+角标
└──────────────────┘          └──────────────────┘
```

| 签到窗口控制 | 窗口外签到返回明确错误，窗口过期后自动标记缺勤 | 手动测试 + 定时任务触发 |
| 请假联动 | 已批准请假的学生在考勤记录中自动标记为「请假」 | 集成测试：创建请假 → 发起考勤 → 验证关联 |
| 统计准确性 | 出勤率计算与实际记录一致 | 手动构造多种状态记录，验证统计数据 |
