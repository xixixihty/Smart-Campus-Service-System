## 🚨 一、重大 Bug（Critical）

### Bug 1：JWT 过滤器 finally 块导致认证绕过 【安全性-严重】

**文件**: [JwtAuthenticationFilter.java:46-56](../src/main/java/com/hxq/smart_campus/filter/JwtAuthenticationFilter.java#L46-L56)

**问题描述**：  
当 JWT Token 过期或无效时，`catch (JwtException e)` 块中调用了 `response.sendError(401)` 和 `return`，但 `finally` 块依然执行了 `filterChain.doFilter(request, response)`。在 Java 中，`finally` 块会在 `return` 之前执行，因此请求**仍然会继续传递到 Controller 层**，JWT 认证完全被绕过。

```java
} catch (JwtException e) {
    response.sendError(401, "JWT认证失败");
    return;          // ← return 会被 finally 覆盖
} finally {
    try {
        filterChain.doFilter(request, response);  // ← 认证失败后仍放行请求！
    } finally {
        SecurityContext.clear();
    }
}
```

**影响**: 任何携带无效/过期 JWT Token 的请求都能通过认证，直接访问需要认证的接口。

**修复方案**：  
将 `filterChain.doFilter(request, response)` 移到 try 块末尾，catch 中直接 return 不执行 doFilter，finally 中只清空 SecurityContext。

### Bug 2：`LeaveApprovalMapper.updateLeaveRequestStatus` 缺少 `@Param("status")` 【运行时会报错】

**文件**: [LeaveApprovalMapper.java:47-49](../src/main/java/com/hxq/smart_campus/mapper/LeaveApprovalMapper.java#L47-L49)

**问题描述**：  
MyBatis 注解 SQL 中引用了 `#{status}`，但方法参数中只有 `@Param("id")` 和 `@Param("leaveApplyStatusApproved")`，缺少 `@Param("status")`。这会导致 MyBatis 抛出 `BindingException`。

```java
@Update("UPDATE leave_request SET status = #{leaveApplyStatusApproved}, update_time = now() WHERE id = #{id} AND status = #{status}")
int updateLeaveRequestStatus(@Param("id") Long id,
                             @Param("leaveApplyStatusApproved") String leaveApplyStatusApproved);
//                             缺 @Param("status")
```

**调用方**: [LeaveApprovalServiceImpl.java:115](../src/main/java/com/hxq/smart_campus/service/impl/LeaveApprovalServiceImpl.java#L115)

**修复方案**：将 `AND status = #{status}` 改为 `AND status = '待审批'`，因为该方法的意图是 CAS 式的乐观更新。

### Bug 6：`LeaveApprovalServiceImpl.insertLeaveRequest` - Long vs int 比较 【逻辑错误】

**文件**: [LeaveApprovalServiceImpl.java:64-66](../src/main/java/com/hxq/smart_campus/service/impl/LeaveApprovalServiceImpl.java#L64-L66)

**问题描述**：  
`getLastInsertId()` 返回 `Long` 对象，用 `<= 0` 与 int 比较会导致自动拆箱。如果 `id` 为 null 则抛出 NPE。

```java

    throw new BusinessException("请假申请失败");
}
```

**修复方案**：`if (id == null || id <= 0)`

## 🔒 三、安全风险

### Risk 1：阿里云 OSS 密钥硬编码

**文件**: [AliOSSUtils.java:24-31](../src/main/java/com/hxq/smart_campus/utils/AliOSSUtils.java#L24-L31)

**问题描述**：阿里云 OSS 的 `accessKeyId` 和 `accessKeySecret` 直接通过 `@Value` 的默认值硬编码在代码中。这是严重的安全风险，代码一旦泄露到远端仓库，云资源将被恶意利用。

```java
@Value("${aliyun.oss.access-key-id:REDACTED_ACCESS_KEY_ID}")     // 硬编码！
@Value("${aliyun.oss.access-key-secret:REDACTED_ACCESS_KEY_SECRET}")  // 硬编码！
```

### Risk 2：JWT 密钥硬编码

**文件**: [JwtUtils.java:21](../src/main/java/com/hxq/smart_campus/utils/JwtUtils.java#L21)

```java
private static final String SECRET = "hxq-smart-campus-jwt-secret-key-...";
```

### Risk 3：`NoticeServiceImpl.evictNoticeCache` 使用 Redis `KEYS *` 【生产环境隐患】

**文件**: [NoticeServiceImpl.java:406-414](../src/main/java/com/hxq/smart_campus/service/impl/NoticeServiceImpl.java#L406-L414)

使用 `redisTemplate.keys(pattern)` 会在 Redis 中执行全量键扫描（O(N) 复杂度），大量 key 时阻塞 Redis。

**修复方案**：使用 SCAN 命令替代，或维护缓存键列表。

## 🎯 五、代码优化建议

### 1. 消除 `RedisTemplate` 泛型警告

**涉及文件**: `CourseSelectionServiceImpl`, `CourseSelectionPeriodServiceImpl`, `NoticeServiceImpl`, `RedisConfiguration`

多处使用 `RedisTemplate` 不带泛型参数，导致大量 unchecked 转换。建议统一为 `RedisTemplate<String, Object>`。

### 2. 抽取公共的时间转换工具方法

几乎每个 `convertToXxxResponseDTO()` 都重复以下代码，建议统一抽取：

```java
public static LocalDateTime parseDateTime(String dateTimeStr) {
    if (dateTimeStr == null || dateTimeStr.isEmpty()) return null;
    return LocalDateTime.parse(dateTimeStr, DATE_TIME_FORMATTER);
}
```

### 3. 排课冲突检测中加入 excludeId

[CourseScheduleServiceImpl.java:232-234](../src/main/java/com/hxq/smart_campus/service/impl/CourseScheduleServiceImpl.java#L232-L234)

`conflictCheck()` 中 `excludeId` 恒为 null，更新排课时会检测到自身冲突。应当根据场景传入需要排除的排课 ID。

### 4. `create_time/update_time` 字段类型优化

VO/DTO 中的时间字段当前用 `String` 接收，再手动转 `LocalDateTime`。建议直接使用 `LocalDateTime`，让 MyBatis TypeHandler 自动处理。

### 5. 移除未使用的 `TimeConflictUtils.hasTimeConflict` 方法

[TimeConflictUtils.java:16-19](../src/main/java/com/hxq/smart_campus/utils/TimeConflictUtils.java)

该静态方法在所有 Service 中未被使用，`CourseSelectionServiceImpl` 自己维护了同名方法，造成冗余。

### 6. 批量删除方法缺少 `@Transactional`

**涉及文件**: `BookServiceImpl.deleteBook()`, `SeatServiceImpl.deleteSeat()`, `StudentServiceImpl.deleteStudent()`

循环检查 + 批量删除应在事务中执行，否则部分失败时数据不一致。
